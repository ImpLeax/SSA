# Практична робота №6

## Тема: Інструменти налагодження для проблем з пам'яттю

**Мета роботи:** Створити програму з навмисним пошкодженням метаданих купи (heap metadata corruption) та на практиці порівняти, як різні інструменти (стандартний розподільник `glibc`, AddressSanitizer та Valgrind) діагностують цю помилку. Зрозуміти відмінності в їхніх підходах до виявлення вразливостей.

---

## 1. Теоретичні відомості

Керування динамічною пам'яттю в мові C здійснюється розробником вручну. Помилки при роботі з вказівниками часто призводять до пошкодження службових даних (метаданих), які менеджер пам'яті використовує для відстеження вільних і зайнятих блоків. 

Для виявлення таких помилок існують різні інструменти, базове порівняння яких наведено у таблиці:

### Таблиця 1. Порівняння інструментів діагностики пам'яті

| Інструмент | Тип аналізу | Механізм роботи | Переваги | Недоліки |
| :--- | :--- | :--- | :--- | :--- |
| **glibc (базовий)** | Динамічний | Перевіряє цілісність метаданих лише під час викликів `malloc` або `free`. | Не потребує додаткових прапорців компіляції чи ПЗ. | Виявляє помилку "постфактум" (часто у вигляді `Segmentation fault`), коли програма вже впала. |
| **AddressSanitizer (ASan)** | Статичний + Динамічний | Додає інструкції перевірки при компіляції (`-fsanitize=address`) та використовує "тіньову пам'ять" (shadow memory) і "червоні зони" (redzones). | Дуже швидкий. Показує точний рядок коду в момент некоректного доступу до пам'яті. | Вимагає перекомпіляції коду. Не виявляє неініціалізовану пам'ять. |
| **Valgrind (Memcheck)** | Динамічний | Запускає програму у власному віртуальному середовищі (JIT-компіляція), перехоплюючи кожну інструкцію роботи з пам'яттю. | Не вимагає перекомпіляції. Знаходить витоки пам'яті та використання неініціалізованих змінних. | Значно уповільнює виконання програми (у 10-50 разів). |

## 2. Аналіз завдання та механізм помилки

У програмі навмисно створено ситуацію **Heap Buffer Overflow** (переповнення буфера в купі), що призводить до пошкодження метаданих.

1. За допомогою `malloc(24)` виділяються два сусідні блоки пам'яті (`chunk1` та `chunk2`).
2. У пам'яті між ними та навколо них менеджер `glibc` розміщує приховані службові структури (метадані: розмір блоку, прапорці статусу тощо).
3. Функція `memset(chunk1, 'A', 64);` записує 64 байти в блок, який розрахований лише на 24. Це призводить до виходу за межі `chunk1` і перезапису простору, де знаходяться метадані `chunk2` та сам `chunk2`.
4. Коли викликається `free(chunk2)`, менеджер пам'яті намагається прочитати його метадані (щоб повернути блок у список вільних), але замість коректних адрес бачить сміття (символи `'A'`, тобто `0x41414141`). Це спричиняє крах.

## 3. Програмна реалізація

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

int main() {
    printf("Process PID: %d\n", getpid());

    char *chunk1 = (char *)malloc(24);
    char *chunk2 = (char *)malloc(24);

    if (chunk1 == NULL || chunk2 == NULL) {
        fprintf(stderr, "Memory allocation failed\n");
        return 1;
    }

    printf("Chunk 1: %p\n", (void*)chunk1);
    printf("Chunk 2: %p\n", (void*)chunk2);

    printf("Executing buffer overflow...\n");
    // Навмисне переповнення: пишемо 64 байти у блок розміром 24
    memset(chunk1, 'A', 64);

    printf("Attempting to free chunk2...\n");
    // Крах станеться тут (для glibc), оскільки метадані chunk2 були затерті символами 'A'
    free(chunk2);
    free(chunk1);

    return 0;
}
```

## 4. Результати тестування та діагностика

Сучасний компілятор `gcc` попередив про помилку ще на етапі компіляції (`warning: ‘memset’ writing 64 bytes into a region of size 24`), але дозволив створити виконуваний файл.

### 4.1. Стандартна поведінка (glibc)

```text
Executing buffer overflow...
Attempting to free chunk2...
Segmentation fault (core dumped)
```
**Аналіз:** Крах (`Segmentation fault`) стався не в момент запису некоректних даних (`memset`), а пізніше — під час виклику `free(chunk2)`. Це найгірший сценарій для розробника, оскільки місце падіння програми не збігається з місцем реальної помилки. `glibc` просто зрозумів, що структури купи зруйновані.

### 4.2. AddressSanitizer (ASan)

```text
=================================================================
==794==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x503000000058...
WRITE of size 64 at 0x503000000058 thread T0
    #0 0x71be332fa97c in memset ...
    #1 0x5be8c8b473a5 in main /home/impleax/SSA/pz6/zd_v2.c:21
...
0x503000000058 is located 0 bytes after 24-byte region [0x503000000040,0x503000000058)
```
**Аналіз:** ASan відпрацював ідеально. Програма була екстрено перервана **саме в момент виконання `memset`** (рядок 21), ще до виклику `free`. ASan оточив виділені 24 байти "червоними зонами" (`fa fa` в shadow bytes), доступ до яких заборонений. Як тільки `memset` торкнувся цих зон, ASan миттєво заблокував операцію і видав точний бектрейс.

### 4.3. Valgrind (Memcheck)

```text
==799== Invalid write of size 8
==799==    at 0x4852CD3: memset (in /usr/libexec/valgrind/vgpreload_memcheck-amd64-linux.so)
==799==    by 0x1092E1: main (zd_v2.c:21)
==799==  Address 0x4a75498 is 0 bytes after a block of size 24 alloc'd
==799==    at 0x4846828: malloc (in /usr/libexec/valgrind/vgpreload_memcheck-amd64-linux.so)
==799==    by 0x109239: main (zd_v2.c:9)
```
**Аналіз:** Valgrind підмінив стандартний `malloc`/`free` на власні реалізації і покроково аналізував виконання. Він виявив `Invalid write` точно на 21-му рядку. Цікаво, що через специфічну організацію пам'яті всередині віртуальної машини Valgrind програма не впала з `Segmentation fault` при `free` (на відміну від `glibc`), оскільки Valgrind ізолює блоки інакше, але успішно зафіксував факт незаконного доступу до пам'яті.

## 5. Загальні висновки

1. **Пошкодження метаданих купи (Heap Corruption)** — це "тиха" помилка. Базовий розподільник `glibc` не помічає її в момент виникнення, а падає пізніше (при алокації або звільненні), що робить налагодження вкрай складним без спецзасобів.
2. **ASan (AddressSanitizer)** є найкращим вибором для активної розробки. Він працює швидко і точно вказує на місце виходу за межі пам'яті завдяки концепції тіньової пам'яті.
3. **Valgrind** є потужним аналізатором, який не потребує перекомпіляції коду. Хоча він сильно уповільнює виконання, його здатність відстежувати кожен байт робить його незамінним інструментом для комплексного аудиту пам'яті у вже скомпільованих бінарних файлах.
