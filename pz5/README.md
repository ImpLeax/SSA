# Практична робота №5

## Тема: Про помилки роботи з пам’яттю в Linux та засоби їх усунення

**Варіант:** 2

Метою роботи було дослідити типові помилки керування пам’яттю в ОС Linux, змоделювати ситуацію цілочисельного переповнення при виклику `malloc()` та проаналізувати наслідки пошкодження метаданих купи (heap) за допомогою інструменту **Valgrind**.

---

## 1. Теоретичні відомості

Робота з пам'яттю в C/C++ вимагає ручного керування, що часто призводить до багів, які важко відстежити. Основні типи проблем наведені у таблиці нижче:

### Таблиця 1. Основні терміни та проблеми

| № | Термін | Значення |
| --- | --- | --- |
| 1 | **Uninitialized variables** | Використання змінної до надання їй значення (читання "сміття"). |
| 2 | **Out-of-bounds access** | Звернення до пам’яті поза межами виділеного масиву. |
| 3 | **Memory leakage** | Втрата доступу до виділеної пам’яті без її звільнення. |
| 4 | **Undefined behavior (UB)** | Стан програми, результат якого не визначений стандартом мови. |
| 5 | **Fragmentation** | Дроблення пам’яті на малі блоки, що заважає великим алокаціям. |
| 6 | **Dangling pointer** | Покажчик на ділянку пам’яті, яка вже була звільнена. |
| 7 | **Double Free** | Повторна спроба звільнити одну й ту саму адресу. |

## 2. Аналіз завдання Варіанту №2

**Завдання:** Створити приклад, де переповнення цілочисельного розміру призводить до виділення занадто малого блоку і подальшого тихого пошкодження heap-метаданих.

### Механізм помилки (Математичне обґрунтування)

В основі помилки лежить обмеження розрядності типу `size_t`. На 64-бітній архітектурі `size_t` має розмір 8 байтів (максимум $2^{64}-1$).

У коді виконується наступний розрахунок:

* `num_elements` = $0x4000000000000001$ (що дорівнює $2^{62} + 1$).
* `element_size` = $4$ байта (розмір `int`).

При обчисленні `total_size = num_elements * element_size` відбувається наступне:


$$(2^{62} + 1) \cdot 4 = 2^{64} + 4$$

Оскільки результат перевищує $2^{64}$, стається переповнення, і в змінну записується лише залишок:


$$2^{64} + 4 \equiv 4 \pmod{2^{64}}$$

В результаті `malloc` отримує запит на виділення всього **4 байтів**, хоча логіка програми передбачає роботу з величезним масивом.

---

## 3. Програмна реалізація

```c
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>

int main() {
    // Величезне число, яке при множенні на 4 викличе переповнення 64-бітного size_t
    size_t num_elements = 0x4000000000000001; 
    size_t element_size = 4;

    size_t total_size = num_elements * element_size;

    printf("Requested number of elements: %zu\n", num_elements);
    printf("Calculated size for malloc (after overflow): %zu bytes\n\n", total_size);

    int *buffer = (int *)malloc(total_size);
    if (buffer == NULL) {
        printf("Memory allocation failed!\n");
        return 1;
    }

    printf("Memory successfully allocated (but the block is too small)!\n");

    // Записуємо 4 елементи (16 байтів) у блок розміром 4 байти
    for (int i = 0; i < 4; i++) {
        buffer[i] = 0xDEADBEEF;
    }

    printf("Data written. Heap metadata corrupted!\n");

    free(buffer); // Тут може статися крах через пошкодження службових даних malloc

    printf("Program successfully completed.\n");
    return 0;
}

```

## 4. Результати тестування

### 4.1. Звичайний запуск

Програма відпрацювала "успішно" в терміналі, але це і є ознакою "тихої" помилки:

```text
Requested number of elements: 4611686018427387905
Size of one element: 4 bytes
Calculated size for malloc (after overflow): 4 bytes

Memory successfully allocated (but the block is too small)!
Data written. Heap metadata corrupted!
Program successfully completed (this line is unlikely to execute).

```

**Висновок:** Без спеціальних інструментів ми не бачимо, що перезаписали пам'ять, яка нам не належить.

### 4.2. Аналіз за допомогою Valgrind

Valgrind чітко детектує проблему **Invalid write**:

```text
==1491== Invalid write of size 4
==1491==    at 0x109282: main (variant_2.c:26)
==1491==  Address 0x4a75484 is 0 bytes after a block of size 4 alloc'd
==1491==    at 0x4846828: malloc (in /usr/libexec/valgrind/vgpreload_memcheck-amd64-linux.so)
==1491==
==1491== ERROR SUMMARY: 3 errors from 1 contexts

```

> **Важливо:** Valgrind вказує, що запис відбувся відразу після (`0 bytes after`) блоку розміром 4 байти. Це класичний **Heap Buffer Overflow**.

## Висновки

1. **Підступність переповнення:** Цілочисельне переповнення (`Integer Overflow`) при розрахунку розміру буфера є критичною вразливістю. Система "чесно" виділяє ту кількість байтів, яка вийшла після переповнення, що призводить до виходу за межі масиву.
2. **Пошкодження метаданих:** Запис за межі виділеного блоку псує службові структури `malloc` (метадані), що знаходяться поруч у купі. Це може призвести до краху програми не в момент запису, а пізніше — під час виклику `free()` або наступного `malloc()`.
3. **Роль Valgrind:** У складних системах такі помилки можуть роками залишатися непоміченими ("тихе пошкодження"). Valgrind є обов'язковим інструментом для перевірки цілісності роботи з пам'яттю, оскільки він відстежує кожен байт і кожну аллокацію.
