# Практична робота №4

## Завдання 4.1. Дослідження лімітів виділення пам’яті malloc(3) на різних архітектурах

Метою завдання було порівняти поведінку функції `malloc()` у 64-бітному та 32-бітному середовищах, а також з'ясувати причини теоретичних та практичних обмежень адресації.

### 1. Теоретичне обґрунтування

Обсяг пам'яті, який може адресувати програма, напряму залежить від розміру типу `size_t`, що визначається архітектурою:

* **x86_64 (64-bit):** `size_t` — 8 байтів. Теоретичний ліміт — 16 ексабайтів ($2^{64}$).
* **x86 (32-bit):** `size_t` — 4 байти. Теоретичний ліміт — 4 гігабайти ($2^{32}$).

Важливим є обмеження `PTRDIFF_MAX` (максимальне значення знакової різниці покажчиків). Для 64-біт це $\approx 8$ ексабайтів, для 32-біт — $\approx 2$ гігабайти.

---

### 2. Компіляція та вихідний код

Для тестування було використано один і той самий код, але скомпільований двома способами:

1. Стандартна 64-бітна збірка.
2. 32-бітна збірка з використанням прапорця `-m32`.

При компіляції 32-бітної версії виникло попередження про **overflow**, оскільки значення 1 ТБ не вміщується у 4-байтну змінну `size_t`.

```bash
gcc -Wall zd1.c -o prog_64

gcc -m32 -Wall zd1.c -o prog_32

```

---

### 4. Аналіз аномалій 32-бітної версії

У 32-бітному режимі програма показала результати, які на перший погляд здаються помилковими, але мають чітке технічне пояснення:

1. **"Успішне" виділення 1 ТБ:**
Через те, що `size_t` у 32-бітній системі обмежений 4 гігабайтами, число `1024ULL * 1024 * 1024 * 1024` (1 ТБ) при спробі записати його в `size_t one_tb` втрачає старші біти. Залишається лише нуль. Виклик `malloc(0)` за стандартом C може повертати або `NULL`, або валідний унікальний покажчик. У нашому випадку система повернула покажчик, тому програма повідомила про успіх.
2. **Успіх із `PTRDIFF_MAX` (~2 ГБ):**
У 32-бітному просторі користувача (User Space) зазвичай доступно близько 2-3 ГБ віртуальної пам'яті (залежить від налаштувань ядра, наприклад, 3G/1G split). Оскільки `malloc` виділяє **віртуальну** пам'ять (Lazy Allocation), система дозволила "зарезервувати" 2 ГБ адресного простору, хоча фізичної пам'яті могло бути менше.
3. **Чому 8 EB, а не 16 EB на 64-біт?**
Теоретично $2^{64}$ — це 16 EB. Проте тип `ptrdiff_t` є знаковим. Стандарт мови C вимагає, щоб різниця між двома покажчиками в межах одного масиву була коректно представлена цим типом. Таким чином, розмір будь-якого об'єкта в пам'яті (масиву, структури) обмежений `PTRDIFF_MAX`, що для 64-бітної системи дорівнює 8 EB ($2^{63}-1$).

### Висновки до завдання 4.1

1. **Архітектурна залежність:** Одна і та ж програма веде себе радикально інакше залежно від розрядності системи. На 32-бітах ми стикаємося з ризиком переповнення цілих чисел при розрахунку обсягів пам'яті.
2. **Цілісність даних:** Помилка в 32-бітній версії (1 ТБ → 0) демонструє, чому важливо використовувати типи на кшталт `uint64_t` або перевіряти переповнення перед виділенням пам'яті.
3. **Віртуальна пам'ять:** Успіх `malloc` не означає, що пам'ять фізично існує; він лише підтверджує, що в адресному просторі процесу знайшовся неперервний блок відповідного розміру.

## Завдання 4.2. Дослідження передачі від’ємного аргументу та переповнення при виклику malloc(3)

Метою завдання було дослідити поведінку функції `malloc()` при передачі їй результату множення, що призводить до цілочисельного переповнення (Integer Overflow), та проаналізувати, як від’ємні значення зі знаком інтерпретуються як беззнакові обсяги пам’яті.

### 1. Теоретичний аналіз проблеми

Функція `malloc(size_t size)` очікує аргумент типу `size_t` (беззнакове ціле). Якщо передати їй від'ємне число (тип `signed`), воно буде автоматично приведено до `unsigned`.

У нашому прикладі:

1. Ми множимо два числа $50\,000 \times 50\,000$. Математичний результат: $2\,500\,000\,000$.
2. Максимальне значення для 32-бітного знакового `int32_t` становить $2\,147\,483\,647$.
3. Результат $2.5$ млрд не вміщується в `int32_t`, що призводить до переповнення. В бінарному вигляді це значення інтерпретується як від’ємне число: **-1,794,967,296**.

### 2. Програмна реалізація

Код імітує ситуацію, де обчислення обсягу пам'яті відбувається з помилкою переповнення.

**Код програми (`zd2.c`):**

```c
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>

int main() {
    int32_t xa = 50000;
    int32_t xb = 50000;

    int32_t num = xa * xb;

    printf("xa = %d\n", xa);
    printf("xb = %d\n", xb);
    printf("num (signed int32_t) = %d\n", num);
    printf("num (cast to size_t) = %zu\n\n", (size_t)num);

    printf("Calling malloc...\n");
    void *ptr = malloc(num);

    if (ptr == NULL) {
        printf("malloc failed to allocate memory.\n");
    } else {
        printf("malloc succeeded.\n");
        free(ptr);
    }

    return 0;
}

```

### 4. Аналіз результатів та відмінностей

Хоча в обох випадках `malloc` повернув помилку (`NULL`), причини та інтерпретація чисел відрізняються:

| Архітектура | Значення `num` (size_t) | Чому такий результат? | Причина відмови `malloc` |
| --- | --- | --- | --- |
| **x86 (32-bit)** | $2\,500\,000\,000$ (2.5 GB) | Від'ємне число `int32_t` просто інтерпретується як `uint32_t`. | У 32-бітній системі користувацький простір обмежений 3 ГБ. Знайти **неперервний** блок у 2.5 ГБ майже неможливо. |
| **x86_64 (64-bit)** | $18.4$ EB | Відбулося **розширення знаку** (Sign Extension) при приведенні з 32-бітного `int` до 64-бітного `size_t`. | Запитано астрономічний обсяг пам'яті (майже 16 ексабайтів), що фізично неможливо виділити. |

**Щодо знакового розширення (64-bit):** Коли ми перетворюємо від'ємний 32-бітний `int` у 64-бітний `size_t`, система заповнює всі нові старші 32 біти одиницями (щоб зберегти "від'ємність" у знаковому представленні). В беззнаковому представленні (`size_t`) це перетворюється на число, близьке до максимуму ($2^{64}-1$).

### Висновки до завдання 4.2

1. **Небезпека `signed` типів:** Використання знакових цілих чисел для підрахунку обсягів пам'яті є грубою помилкою, оскільки переповнення може призвести до передачі в `malloc` неочікувано великих значень.
2. **Механізм приведення:** Якщо `malloc` отримує від'ємний аргумент, він інтерпретує його як дуже велике додатне число.
3. **Архітектурна різниця:** На 64-бітних системах через розширення знаку помилка стає ще масштабнішою (від мегабайтів/гігабайтів до ексабайтів).
4. **Стабільність:** `malloc` надійно обробляє такі ситуації, повертаючи `NULL`, але програма повинна бути готова до такої відповіді, щоб не виникло звернення за нульовим покажчиком.

## Завдання 4.3. Дослідження поведінки malloc(0)

Метою завдання було з'ясувати, як функція `malloc()` обробляє запит на виділення 0 байтів пам'яті, та перевірити коректність отриманого покажчика через утиліту трасування `ltrace`.

### 1. Теоретичне обґрунтування

Згідно зі стандартом C (ISO/IEC 9899), поведінка `malloc(0)` є **implementation-defined** (визначається реалізацією). Бібліотека може:

1. Повернути `NULL`.
2. Повернути унікальний покажчик, який можна безпечно передати у `free()`.

У середовищі Linux (бібліотека **glibc**) виклик `malloc(0)` зазвичай повертає покажчик на найменший можливий блок пам'яті в купі (heap), щоб забезпечити унікальність адреси для кожного виклику.

### 2. Програмна реалізація

Програма виконує виклик `malloc(0)`, перевіряє результат і звільняє пам'ять.

**Код програми (`zd3.c`):**

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    void *ptr = malloc(0);

    if (ptr == NULL) {
        printf("malloc(0) returned NULL.\n");
    } else {
        printf("malloc(0) returned a non-NULL pointer: %p\n", ptr);
        free(ptr);
        printf("Pointer successfully freed.\n");
    }

    return 0;
}

```

---

### 3. Результати виконання та трасування

Спершу було виконано стандартний запуск програми, а потім — аналіз через `ltrace` для відстеження викликів динамічних бібліотек.

**Вивід звичайного запуску:**

```bash
impleax@Impleax:~/SSA/pz4/zd3$ gcc -Wall zd3.c -o prog
impleax@Impleax:~/SSA/pz4/zd3$ ./prog
malloc(0) returned a non-NULL pointer: 0x5d66e58552a0
Pointer successfully freed.

```

**Аналіз через `ltrace`:**
Ця утиліта дозволяє побачити реальні аргументи та значення, що повертаються функціями бібліотеки `libc`.

```bash
impleax@Impleax:~/SSA/pz4/zd3$ ltrace ./prog
malloc(0)                                          = 0x5ec021d762a0
printf("malloc(0) returned a non-NULL po"..., 0x5ec021d762a0) = 54
free(0x5ec021d762a0)                               = <void>
puts("Pointer successfully freed.")                = 28
+++ exited (status 0) +++

```

---

### 4. Пояснення поведінки програми

1. **Результат `malloc(0)`:** Як видно з логу `ltrace`, виклик `malloc(0)` не повернув `0` (NULL), а повернув реальну адресу в пам'яті (`0x5ec0...`).
2. **Мінімальний чанк:** Бібліотека `glibc` виділяє мінімальний службовий блок (chunk). Це необхідно для того, щоб підтримувати цілісність структури купи та дозволяти порівнювати покажчики.
3. **Оптимізація виводу:** Цікаво, що в `ltrace` другий виклик `printf` відображається як `puts`. Це стандартна оптимізація компілятора `gcc`: якщо рядок не містить специфікаторів формату (як `%d`), він замінює важку функцію `printf` на швидшу `puts`.
4. **Коректність `free()`:** Трасування підтверджує, що `free()` отримує ту саму адресу, яку повернув `malloc`. Це не призводить до помилок (segmentation fault), що підтверджує валідність отриманого покажчика для системних механізмів керування пам'яттю.

### Висновки до завдання 4.3

1. **Специфіка Linux:** У системі Ubuntu/WSL виклик `malloc(0)` завжди повертає валідну адресу в купі, а не `NULL`.
2. **Роль `ltrace`:** Інструмент дозволив переконатися, що програма взаємодіє з купою стандартним чином, і навіть "нульовий" запит створює повноцінний об'єкт керування пам'яттю.
3. **Практичне значення:** Програміст повинен пам'ятати, що навіть якщо `malloc(0)` повернув не-NULL, записувати дані за цією адресою не можна, оскільки розмір виділеного корисного навантаження дорівнює 0.

## Завдання 4.4. Аналіз dangling pointers та помилок Double Free

Метою завдання було проаналізувати логічну помилку в циклічному управлінні пам'яттю, продемонструвати наслідки некоректного звільнення ресурсів та реалізувати безпечний варіант коду.

### 1. Аналіз помилки у наданому коді

У запропонованому фрагменті коду є дві критичні проблеми:

1. **Dangling Pointer (Зависший покажчик):** Після виконання `free(ptr)` значення самого покажчика (адреса в пам'яті) не змінюється. Він продовжує вказувати на ту саму ділянку пам'яті, яка вже повернута операційній системі.
2. **Помилкова логіка перевірки:** На другій ітерації циклу умова `if (!ptr)` виявляється **хибною**, оскільки покажчик не є `NULL`. Програма пропускає виклик `malloc(n)`, але намагається використовувати «стару» адресу та, що найгірше, викликає `free(ptr)` вдруге для тієї ж адреси.

---

### 2. Тестові випадки: Flawed vs Correct

Я реалізував два сценарії, щоб наочно побачити різницю в поведінці системи.

#### А. Некоректний варіант (`inncorrect.c`)

Тут ми залишаємо покажчик "брудним" після виклику `free`.

```c
// ... фрагмент циклу ...
while (condition > 0) {
    if (!ptr) ptr = malloc(1024); // Виконається лише на 1-й ітерації
    free(ptr); // На 2-й ітерації призведе до Double Free
    condition--;
}

```

#### Б. Правильний варіант (`correct.c`)

Після звільнення пам'яті ми явно присвоюємо покажчику значення `NULL`. Це гарантує, що наступна перевірка `if (!ptr)` спрацює коректно.

```c
// ... безпечний підхід ...
free(ptr);
ptr = NULL; // Тепер покажчик знову "чистий"

```

---

### 3. Результати експерименту та реакція системи

Результати в терміналі чітко показують, як сучасні інструменти розробки допомагають виявляти такі помилки.

**1. Реакція компілятора (GCC):**
GCC помітив проблему ще на етапі збірки, видавши попередження про використання покажчика після звільнення:
`warning: pointer ‘ptr’ used after ‘free’ [-Wuse-after-free]`

**2. Реакція Runtime (Runtime Crash):**
При запуску `inncorrect` спрацював механізм захисту бібліотеки `glibc`:

```bash
impleax@Impleax:~/SSA/pz4/zd4$ ./inncorrect
Starting flawed loop...
Allocated memory at: 0x610171ed26b0
Using memory at: 0x610171ed26b0
Freed memory at: 0x610171ed26b0
Using memory at: 0x610171ed26b0
free(): double free detected in tcache 2
Aborted (core dumped)

```

**Аналіз виводу:**

* На другій ітерації програма використала ту саму адресу `0x610171ed26b0`.
* Система виявила спробу повторного звільнення тієї ж ділянки в `tcache` (швидкий кеш аллокатора) і негайно завершила процес, щоб запобігти пошкодженню пам'яті.

**3. Успішне виконання (`correct`):**
У виправленій версії на кожній ітерації відбувається новий цикл виділення-звільнення, що є абсолютно безпечним.

---

### Висновки до завдання 4.4

1. **Золоте правило:** Після виклику `free(ptr)` завжди слід додавати `ptr = NULL;`, якщо цей покажчик може бути використаний далі в коді. Це перетворює потенційну помилку "Double Free" або "Use-after-free" на передбачуваний Segfault (якщо ви спробуєте розіменувати NULL) або дозволяє коректно відпрацювати логічним перевіркам.
2. **Діагностика:** Сучасні версії `glibc` мають вбудовані механізми виявлення Double Free, проте покладатися лише на них не можна, оскільки в складних програмах такі помилки можуть призводити до вразливостей безпеки.
3. **Логіка циклів:** Управління ресурсами всередині циклів вимагає особливої уваги до стану змінних-покажчиків на початку та в кінці кожної ітерації.

## Завдання 4.5. Дослідження поведінки realloc(3) при вичерпанні ресурсів

Метою завдання було з'ясувати, що відбувається з оригінальним блоком пам'яті, якщо функція `realloc()` не може задовольнити запит на розширення, та продемонструвати безпечний спосіб оновлення покажчиків.

### 1. Теоретичне обґрунтування

Функція `void *realloc(void *ptr, size_t size)` має специфічну логіку обробки помилок:

* **При успіху:** вона виділяє новий блок, копіює дані, звільняє старий блок (`ptr`) і повертає нову адресу.
* **При невдачі:** вона повертає `NULL`, але **не звільняє** старий блок пам'яті.

### 2. Програмна реалізація

Для гарантованого провокування помилки ми намагаємося розширити блок до `SIZE_MAX` (максимально можливе значення для архітектури).

**Код програми (`zd5.c`):**

```c
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>

int main() {
    void *ptr = malloc(1024);

    if (ptr == NULL) {
        printf("Initial malloc failed.\n");
        return 1;
    }

    printf("Original memory allocated at: %p\n", ptr);

    // Використання тимчасового покажчика для безпеки
    void *temp = realloc(ptr, SIZE_MAX);

    if (temp == NULL) {
        printf("realloc failed to allocate SIZE_MAX. Returned NULL.\n");
        printf("Original pointer (ptr) is still valid: %p\n", ptr);

        free(ptr); // Ми все ще можемо звільнити пам'ять!
        printf("Original memory successfully freed.\n");
    } else {
        printf("realloc succeeded unexpectedly.\n");
        ptr = temp;
        free(ptr);
    }

    return 0;
}

```

### 4. Висновки до завдання 4.5

1. **Цілісність даних:** Експеримент підтвердив, що у разі помилки `realloc()` старий покажчик `ptr` залишається валідним, а дані за цією адресою не пошкоджуються.
2. **Важливість `temp` покажчика:** Використання проміжної змінної при виклику `realloc` є обов'язковим стандартом безпечного програмування на C. Це єдиний спосіб уникнути витоку ресурсів при роботі в умовах обмеженої пам'яті.
3. **Системні ліміти:** Повернення `NULL` при запиті на `SIZE_MAX` є логічною поведінкою системи, оскільки такий обсяг перевищує будь-які фізичні та віртуальні можливості сучасної архітектури.
